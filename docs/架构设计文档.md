# Enterprise RAG Assistant 架构设计文档

## 1. 文档目的

本文档描述 `Enterprise RAG Assistant` 当前版本的系统架构、模块职责、核心链路、数据模型和后续演进方向，用于开发协作、部署排查、功能扩展和项目讲解。

当前项目已完成企业知识库 RAG MVP：前端使用 Next.js，AI 服务使用 FastAPI，知识库和会话数据使用 PostgreSQL + pgvector 持久化，模型侧接入 GLM Embedding 与 GLM Chat，并提供普通问答和 SSE 流式问答接口。

## 2. 系统定位

`Enterprise RAG Assistant` 是面向企业内部知识库场景的智能问答平台。系统通过上传企业制度、产品手册、FAQ、培训资料等文档，完成文本解析、切分、向量化、入库、相似度检索和大模型生成，让用户可以基于企业私有资料获得带引用来源的回答。

系统当前聚焦四类能力：

- 知识库管理：上传文档、解析文档、切分文本、生成向量、保存文档与片段。
- RAG 问答：根据问题检索相关片段，构造 Prompt，调用模型生成答案。
- 引用溯源：返回文档名、片段编号、相似度分数和引用摘要。
- 会话体验：支持多轮会话 ID、普通问答接口和 SSE 流式输出。

## 3. 当前技术栈

| 层级 | 技术 | 当前职责 |
| --- | --- | --- |
| 前端应用 | Next.js、React、TypeScript、Tailwind CSS | 企业知识库控制台、智能问答界面、文档上传、来源展示、服务状态展示 |
| AI 服务 | FastAPI | 对外提供文档入库、知识库查询、RAG 问答、流式问答和健康检查接口 |

| 本地文件 | data/uploads | 保存上传原始文件，支持重建知识库 |
| 测试 | pytest | 覆盖 AI 服务核心逻辑 |

## 6. 后端服务设计

### 6.1 FastAPI 入口

`backend/ai_service/main.py` 创建 FastAPI 应用，配置 CORS，并挂载 `api/routes.py` 中的路由。

当前 CORS 为开放配置，方便本地开发。生产环境应按前端域名收敛 `allow_origins`。

### 6.2 API 路由

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/health` | 健康检查，返回服务版本和数据目录 |
| POST | `/api/documents/upload` | 上传文档并立即入库 |
| GET | `/api/documents` | 获取知识库文档列表 |
| DELETE | `/api/documents/{document_id}` | 删除文档及其片段 |
| POST | `/api/knowledge/rebuild` | 基于 `data/uploads` 重建知识库 |
| POST | `/api/chat/ask` | 普通 RAG 问答 |
| POST | `/api/chat/stream` | SSE 流式 RAG 问答 |
| GET | `/api/chat/conversations/{conversation_id}` | 获取指定会话历史 |

### 6.3 知识库入库服务

`KnowledgeService` 负责文档入库编排：

1. 校验文件扩展名。
2. 读取上传内容并计算 MD5。
3. 将原始文件保存到 `data/uploads`。
4. 调用 `document_loader.py` 提取文本。
5. 调用 `text_splitter.py` 按 `CHUNK_SIZE` 和 `CHUNK_OVERLAP` 切分。
6. 调用向量存储服务写入文档、片段、元数据和 embedding。

当前支持的扩展名由 `SUPPORTED_EXTENSIONS` 控制，包括 `.txt`、`.md`、`.csv`、`.json`、`.pdf`。

### 6.4 向量存储服务

当前默认使用 `PostgresVectorStore`：

- 启动时自动执行 `CREATE EXTENSION IF NOT EXISTS vector`。
- 自动创建 `documents` 和 `document_chunks` 表。
- 文档入库时调用 GLM Embedding 生成向量。
- 查询时对用户问题生成向量，并使用 pgvector 的 `<=>` 距离进行相似度检索。
- 通过 `VECTOR_SCORE_THRESHOLD` 控制最低召回分数。

代码中仍保留 `LocalVectorStore`，可作为早期本地 JSON 索引方案或离线开发参考；当前生产路径由 `storage_factory.py` 指向 PostgreSQL。

### 6.5 RAG 问答服务

`RAGService` 是问答主链路，负责：

1. 接收问题、会话 ID、召回数量和回答模式。
2. 调用向量库检索相关片段。
3. 将检索结果转换为引用来源结构。
4. 构造系统提示词和用户提示词。
5. 调用 GLM Chat 生成答案。
6. 在模型不可用时退回本地模板回答。
7. 写入会话历史。
8. 返回答案、来源、会话 ID、回答模式和模型名。

回答模式：

| 模式 | 请求值 | 默认模型 | 适用场景 |
| --- | --- | --- | --- |
| 快速模式 | `fast` | `GLM-4V-Flash` | 常规制度问答、简单事实查询 |
| 思考模式 | `thinking` | `GLM-4.1V-Thinking-Flash` | 多条件问题、跨片段归纳、较复杂解释 |

### 6.6 流式问答

`/api/chat/stream` 使用 SSE 返回事件。后端事件格式统一为 JSON，并以 `data: ...\n\n` 输出。

事件类型：

| 类型 | 内容 |
| --- | --- |
| `answer_delta` | 模型实时生成的文本片段 |
| `sources` | 本次回答引用的知识库片段 |
| `done` | 会话 ID、回答模式和模型信息 |

前端在 `frontend/next-web/lib/api.ts` 中通过 `ReadableStream` 读取响应体，并使用 `consumeSseBuffer` 解析 SSE 数据。

## 7. 数据模型

### 7.1 documents

保存文档级元数据。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | uuid | 文档 ID |
| name | text | 原始文档名 |
| metadata | jsonb | 文件路径、MD5、扩展名等元数据 |
| created_at | timestamptz | 入库时间 |

### 7.2 document_chunks

保存文本片段和向量。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | uuid | 片段 ID |
| document_id | uuid | 所属文档 ID |
| document_name | text | 冗余文档名，便于结果返回 |
| chunk_index | integer | 文档内片段序号 |
| content | text | 片段正文 |
| metadata | jsonb | 片段继承的文档元数据 |
| created_at | timestamptz | 创建时间 |
| embedding | vector | GLM embedding 向量 |
| embedding_model | text | 向量模型名 |

索引：

- `idx_document_chunks_document_id`
- pgvector 距离检索当前直接基于 `embedding <=> query_vector` 排序，后续可增加 ivfflat 或 hnsw 索引。

### 7.3 chat_turns

保存多轮会话历史。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | uuid | 单轮问答 ID |
| conversation_id | uuid | 会话 ID |
| question | text | 用户问题 |
| answer | text | 模型回答 |
| sources | jsonb | 引用来源快照 |
| created_at | timestamptz | 创建时间 |

索引：

- `idx_chat_turns_conversation_id`

## 8. 核心链路

### 8.1 文档入库链路

```text
用户选择文件
  -> Next.js 调用 /api/documents/upload
  -> FastAPI 接收 UploadFile
  -> 校验扩展名
  -> 保存原始文件到 data/uploads
  -> 解析文本
  -> 文本切分
  -> 调用 GLM Embedding
  -> 写入 documents
  -> 写入 document_chunks
  -> 返回 document_id 和 chunk_count
  -> 前端刷新文档列表
```

### 8.2 普通问答链路

```text
用户提问
  -> Next.js 调用 /api/chat/ask
  -> FastAPI 进入 RAGService.ask
  -> 问题向量化
  -> pgvector 相似度检索
  -> 构造 sources
  -> 构造 Prompt
  -> 调用 GLM Chat
  -> 写入 chat_turns
  -> 返回 answer + sources + conversation_id
```

### 8.3 流式问答链路

```text
用户提问
  -> Next.js 调用 /api/chat/stream
  -> FastAPI 进入 RAGService.stream_ask
  -> 问题向量化和片段检索
  -> 调用 GLM Chat stream=true
  -> 持续发送 answer_delta
  -> 发送 sources
  -> 写入 chat_turns
  -> 发送 done
  -> 前端逐字更新回答区域
```

## 9. 前端设计

当前前端是单页企业知识库控制台，主要由三块组成：

- 左侧导航：展示产品身份和服务状态。
- 中间问答区：展示欢迎语、用户问题、助手回答、回答模式和流式生成状态。
- 右侧检查面板：上传文档、查看文档列表、展示最新引用来源。

前端 API 封装集中在 `lib/api.ts`，负责：

- 普通问答 `askQuestion`。
- 流式问答 `streamQuestion`。
- 文档列表 `fetchDocuments`。
- 文档上传 `uploadDocument`。
- 健康检查 `checkHealth`。
- SSE 缓冲区解析 `consumeSseBuffer`。

## 10. 配置项

主要配置集中在 `backend/ai_service/config.py`，并支持从项目根目录 `.env` 读取。

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `DATABASE_URL` | 空 | PostgreSQL 连接地址，当前必需 |
| `GLM_API_KEY` | 空 | GLM API Key |
| `GLM_EMBEDDING_URL` | GLM 官方 embedding 地址 | Embedding API 地址 |
| `GLM_EMBEDDING_MODEL` | `embedding-3` | Embedding 模型 |
| `GLM_CHAT_URL` | GLM 官方 chat completions 地址 | Chat API 地址 |
| `GLM_FAST_MODEL` | `GLM-4V-Flash` | 快速模式模型 |
| `GLM_THINKING_MODEL` | `GLM-4.1V-Thinking-Flash` | 思考模式模型 |
| `CHUNK_SIZE` | `700` | 文本片段长度 |
| `CHUNK_OVERLAP` | `120` | 文本片段重叠长度 |
| `TOP_K` | `4` | 默认召回片段数 |
| `VECTOR_SCORE_THRESHOLD` | `0.20` | 向量召回最低相似度 |

前端配置：

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `NEXT_PUBLIC_API_BASE_URL` | `http://127.0.0.1:8000` | 后端 API 地址 |

## 11. 可靠性与降级策略

当前已经具备的降级能力：

- Chat 模型不可用时，`RAGService` 会退回基于检索片段的本地模板回答。
- 流式生成中断时，会保留已生成内容，并返回错误说明或模板回答。
- 前端健康检查失败时，界面显示后端未就绪。

当前需要注意的限制：

- 使用 PostgreSQL 存储时，文档入库必须配置 `GLM_API_KEY`，因为需要生成 pgvector 检索向量。
- `DATABASE_URL` 当前是必需配置，服务启动时会创建表和扩展。
- CORS 当前开放，生产环境需要收敛。
- 删除文档会级联删除对应片段，但不会删除 `data/uploads` 中的原始文件。

## 12. 安全与权限边界

当前版本是 MVP，还没有实现用户认证、角色权限和租户隔离。进入企业级版本时建议增加：

- 用户登录和 Token 鉴权。
- 文档级权限控制。
- 部门、角色、知识库空间隔离。
- 上传文件大小和类型限制。
- API 访问审计。
- 模型调用日志与敏感信息脱敏。
- 管理员后台与普通用户控制台分离。

规划上可以引入 Django 作为传统业务后台，负责用户、角色、权限、文档元数据、审计日志和管理后台；FastAPI 继续专注 AI 服务能力。

## 13. 可观测性与运维建议

后续建议补齐：

- 请求日志：记录接口耗时、状态码、conversation_id。
- 模型调用日志：记录模型名、耗时、token 用量和失败原因。
- 检索日志：记录 top_k、相似度分数、命中文档。
- 健康检查扩展：增加数据库连通性、GLM API 可用性检查。
- 数据库迁移：用 Alembic 或 Django migration 管理表结构演进。
- 备份策略：定期备份 PostgreSQL 和 `data/uploads`。

## 14. 后续演进路线

### 14.1 检索增强

- 为 pgvector 增加 HNSW 或 IVFFlat 索引。
- 增加关键词检索，与向量检索组成混合检索。
- 增加 Rerank 模型，提高最终上下文质量。
- 增加按文档类型、部门、权限、标签过滤。

### 14.2 文档处理增强

- 增加 `.docx`、`.xlsx`、网页、图片 OCR 支持。
- 增加重复文档检测和版本管理。
- 增加入库任务状态，支持大文件异步处理。
- 增加失败重试和部分片段重建。

### 14.3 企业能力增强

- 接入 Django 用户权限系统。
- 增加知识库空间和多租户隔离。
- 增加操作日志、审计日志和管理员后台。
- 增加会话收藏、导出、标题生成和反馈评价。

### 14.4 Agent 与工作流

- 引入 LangGraph 编排多步骤问答流程。
- 封装企业业务接口为工具，例如订单、库存、工单、报销进度查询。
- 对比 Dify 低代码知识库方案，沉淀自研方案与平台方案的适用边界。
- 评估 CrewAI 多智能体方案，用于复杂任务拆解、答案审核和人工转接。

## 15. 项目讲解摘要

这个项目可以概括为：基于 Next.js、FastAPI、PostgreSQL + pgvector 和 GLM 模型构建的企业知识库 RAG 问答系统。用户上传企业文档后，系统会解析文本、切分片段、生成 embedding 并写入 pgvector；提问时先进行向量检索，再把相关片段交给 GLM Chat 生成带引用的回答。前端提供文档上传、流式问答、引用来源和服务状态展示，后端保留了向 Django 权限后台、混合检索、Rerank、Agent 工作流和 Dify 对照方案演进的空间。
