# 知识库管理页面架构设计

本文档描述知识库管理页面的前后端架构、数据模型、接口扩展和演进方案。设计目标是在当前 FastAPI + PostgreSQL + pgvector + Next.js 的 MVP 基础上，形成可维护、可扩展的企业知识库运维模块。

## 1. 当前基础

当前系统已经具备以下能力：

- 前端 `InspectorPanel` 支持文档上传、文档列表、引用来源和处理链路展示。
- `frontend/next-web/lib/api.ts` 已封装 `fetchDocuments`、`uploadDocument`、`checkHealth`。
- FastAPI 已提供文档上传、列表、删除和重建接口。
- `KnowledgeService` 负责编排文件校验、保存、解析、切分、入库。
- `PostgresVectorStore` 负责 `documents`、`document_chunks` 建表、写入、查询、删除和向量检索。
- 文档 metadata 已保存 `source_path`、`file_md5`、`extension`。

当前短板：

- 知识库能力还嵌在聊天页侧栏，不适合大量文档管理。
- 文档没有独立详情接口和片段列表接口。
- 上传入库是同步过程，缺少任务状态。
- 缺少文档搜索、筛选、分页、单文档重入库。
- 删除和重建没有前端二次确认与审计日志。
- 权限、部门、标签、引用日志还未落库。

## 2. 目标架构

```text
Browser
  |
  | HTTP
  v

  |-- app/knowledge/page.tsx
  |-- components/knowledge/*
  |-- lib/api.ts
  |
  | REST API
  v

  |-- api/routes.py
  |-- services/knowledge_service.py
  |-- services/vector_store_service.py
  |-- services/document_task_service.py       后续新增
  |-- services/audit_service.py               后续新增
  |

  |-- documents
  |-- document_chunks
  |-- document_ingest_tasks                   后续新增
```

第一期继续由 FastAPI 直接承载知识库管理接口。后续如果接入 Django 业务后台，可让 Django 管理用户、角色、部门、审计和文档元数据，FastAPI 保持 AI 入库、Embedding、检索和 RAG 服务职责。

## 4. 后端架构

### 4.1 服务职责

`routes.py`：

- 接收 HTTP 请求
- 参数校验和错误转换
- 调用业务服务
- 返回统一响应

`KnowledgeService`：

- 文件扩展名校验
- 文件保存和 MD5 计算
- 文本解析
- 文本切分
- 调用向量存储写入
- 后续负责创建和更新入库任务状态

`PostgresVectorStore`：

- `documents` 和 `document_chunks` 表维护
- 文档列表查询
- 文档删除
- 文档片段查询
- 向量检索

建议新增 `DocumentTaskService`：

- 记录上传、重入库、重建索引任务
- 管理状态：pending、running、success、failed
- 记录错误信息、耗时、处理数量

建议新增 `AuditService`：

- 记录上传、删除、重建、权限变更
- 后续关联用户和角色

### 4.2 同步与异步边界

当前同步入库适合小文件和演示场景：

```text
upload -> parse -> split -> embedding -> insert -> response
```

企业场景建议演进为异步任务：

```text
upload -> save original file -> create task -> response task_id
worker -> parse -> split -> embedding -> insert -> update task status
frontend -> poll task status
```

第一期可保持同步，但接口响应中应尽量返回 `document_id`、`chunk_count`、`metadata` 和错误信息，为后续异步化保留结构。

## 6. 接口设计

### 6.1 文档列表

```http
GET /api/documents?keyword=&extension=&status=&page=1&page_size=20
```

第一期可先不分页，前端本地筛选。文档量变大后改为后端分页。

响应：

```json
{
  "documents": [
    {
      "document_id": "uuid",
      "document_name": "员工守则制度.md",
      "chunk_count": 12,
      "created_at": "2026-05-22T12:00:00+00:00",
      "metadata": {
        "source_path": "data/uploads/xxx.md",
        "file_md5": "md5",
        "extension": ".md"
      }
    }
  ],
  "total": 1
}
```

### 6.2 文档详情

```http
GET /api/documents/{document_id}
```

返回文档基础信息、metadata、统计和最近任务。

### 6.3 文档片段

```http
GET /api/documents/{document_id}/chunks?page=1&page_size=50
```

响应：

```json
{
  "document_id": "uuid",
  "chunks": [
    {
      "chunk_id": "uuid",
      "chunk_index": 0,
      "content": "片段内容",
      "created_at": "2026-05-22T12:00:00+00:00",
      "embedding_model": "embedding-3"
    }
  ]
}
```

### 6.4 删除文档

```http
DELETE /api/documents/{document_id}
```

当前实现会删除 `documents` 并通过外键级联删除 `document_chunks`，但不会删除 `data/uploads` 原始文件。页面文案应明确这一点，避免用户误以为源文件也被移除。

### 6.5 重建知识库

```http
POST /api/knowledge/rebuild
```

当前实现：

- 清空 `documents` 和 `document_chunks`
- 遍历 `data/uploads`
- 重新解析和写入向量库

风险：

- 同步重建期间接口可能阻塞
- 中途失败可能导致部分数据缺失
- 缺少任务状态和回滚

建议二期改为任务化。

### 6.6 检索测试

```http
POST /api/knowledge/search
```

请求：

```json
{
  "query": "报销多久可以打款？",
  "top_k": 5,
  "score_threshold": 0.2,
  "filters": {
    "extension": ".md"
  }
}
```

响应复用 RAG `Source` 结构，方便前端展示命中文档、片段和分数。

## 7. 入库流程

```text
用户选择文件
  -> 前端校验扩展名和大小
  -> POST /api/documents/upload
  -> FastAPI 接收 UploadFile
  -> KnowledgeService 校验扩展名
  -> 计算 MD5
  -> 保存到 data/uploads
  -> document_loader 提取文本
  -> text_splitter 切分片段
  -> GLMEmbeddingClient 生成向量
  -> PostgresVectorStore 写 documents
  -> PostgresVectorStore 写 document_chunks
  -> 返回 document_id、chunk_count
  -> 前端刷新列表和统计
```

失败处理：

- 扩展名不支持：返回 400
- 文本解析失败：返回可读错误
- Embedding 缺少 API Key：返回服务配置错误
- 数据库写入失败：返回入库失败，后续写任务日志

## 8. 删除流程

```text
用户点击删除
  -> 前端弹出确认框
  -> DELETE /api/documents/{document_id}
  -> 后端统计片段数
  -> 删除 documents
  -> 数据库级联删除 document_chunks
  -> 返回 deleted_chunks
  -> 前端刷新列表和统计
```

建议：

- 前端确认框显示文档名和片段数
- 删除后提示“已删除索引，原始上传文件仍保留”
- 后续审计日志记录操作人、时间和结果

## 9. 重建流程

```text
用户点击重建索引
  -> 前端弹出危险操作确认
  -> POST /api/knowledge/rebuild
  -> vector_store.clear()
  -> KnowledgeService.ingest_directory(data/uploads)
  -> 返回重建结果
  -> 前端刷新列表
```

二期任务化流程：

```text
POST /api/knowledge/rebuild
  -> create rebuild task
  -> return task_id
worker
  -> clear old index
  -> ingest files one by one
  -> update progress
frontend
  -> poll /api/knowledge/tasks/{task_id}
```

## 10. 权限设计

第一期暂不强制实现，但页面和接口应按权限边界设计。

| 能力 | 普通员工 | 维护员 | 管理员 |
| --- | --- | --- | --- |
| 查看文档列表 | 可选 | 是 | 是 |
| 查看片段 | 否 | 是 | 是 |
| 上传文档 | 否 | 是 | 是 |
| 删除文档 | 否 | 是 | 是 |
| 重建索引 | 否 | 否 | 是 |
| 查看审计日志 | 否 | 否 | 是 |
| 配置切分和检索参数 | 否 | 否 | 是 |

后续可由 Django 提供用户、角色和 Token 鉴权；FastAPI 校验请求头中的用户身份和权限范围。

## 11. 可观测性

建议记录：

- 上传耗时
- 解析耗时
- 切分片段数
- Embedding 耗时
- 写库耗时
- 失败原因
- 删除和重建操作日志
- 检索测试的 top_k、分数和命中文档

这些数据可以支持后续回答：

- 哪类文档最容易解析失败？
- 哪些文档经常被问答引用？
- 哪些问题召回分数低？
- 知识库是否需要补充资料？

## 12. 风险与处理

| 风险 | 表现 | 建议 |
| --- | --- | --- |
| 同步入库阻塞 | 大文件上传等待很久 | 二期引入任务队列 |
| 重复文件 | 同一资料被多次入库 | 基于 MD5 去重或版本化 |
| 删除误操作 | 资料索引丢失 | 二次确认、审计、软删除 |
| 解析质量差 | PDF 或表格切分不准 | 增加解析器和片段预览 |
| 召回不准 | 聊天页提示知识不足 | 增加检索测试、混合检索、Rerank |
| 权限泄露 | 普通用户看到敏感资料 | 接入 RBAC 和 metadata 过滤 |

## 13. 实施计划

### 阶段一：独立页面

- 新增 `/knowledge` 页面
- 复用现有上传、列表和统计能力
- 增加删除、刷新、重建操作
- 增加搜索和类型筛选
- 增加加载态、错误态、空状态

### 阶段二：详情与片段

- 新增文档详情接口

- 前端增加详情抽屉

### 阶段三：任务化

- 新增 `document_ingest_tasks`
- 上传、重入库、重建索引改为任务

### 阶段四：企业增强

- 接入用户、角色、部门和权限范围
- 增加审计日志
- 增加标签、分类、版本管理
- 增加检索测试、引用日志和质量分析

## 15. 结论

知识库管理页应优先解决“资料可管理、索引可确认、问题可排查”。第一期不要急着做复杂权限和质量分析，先把上传、列表、删除、重建、搜索、详情这些维护员每天会用的动作做顺。等这条主链路稳定，再补任务化、权限、引用日志和检索质量分析，整个 RAG 平台会更像一个可运营的企业系统，而不是只会问答的 Demo。
