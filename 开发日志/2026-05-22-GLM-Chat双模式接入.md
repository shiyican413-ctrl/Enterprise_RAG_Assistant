# 2026-05-22 GLM Chat 双模式接入记录

## 本次做了什么

这次把 RAG 的回答生成从“本地模板拼接”升级为“调用 GLM Chat Completions 生成回答”，并新增了两个回答模式。

主要修改：

- 新增 `backend/ai_service/services/chat_model_service.py`，封装 GLM Chat Completions 调用。
- 在 `backend/ai_service/config.py` 增加 Chat 模型配置。
- 在 `/api/chat/ask` 和 `/api/chat/stream` 请求里支持 `answer_mode`。
- 在 `rag_service.py` 中把检索片段交给聊天模型生成回答。
- 在前端问答区新增“快速 / 思考”模式切换。
- 更新 `README.md` 和 `docs/部署与启动说明.md`，补充双模式配置说明。
- 增加测试，验证思考模式会正确传到 RAG 服务并返回模型信息。

## 两个模式是什么

专业一点说：

- 快速模式：`answer_mode=fast`，调用 `GLM-4V-Flash`。
- 思考模式：`answer_mode=thinking`，调用 `GLM-4.1V-Thinking-Flash`。
- 两个模式都使用 `POST /paas/v4/chat/completions`。
- 后端会先做知识库检索，再把召回片段、用户问题和引用编号要求一起发给模型。
- 模型回答会返回 `answer_mode` 和 `model`，前端可以展示本次使用的模式和模型。

大白话说：

- 以前系统是“找到资料后，把片段按模板拼出来”。
- 现在系统会“先找资料，再让 GLM 根据资料组织一段像样的回答”。
- 快速模式适合简单问题，返回更快。
- 思考模式适合复杂问题，比如跨多个文档、多个条件的分析。
- 回答里会尽量带 `[1] [2]` 这种引用编号，方便知道依据来自哪段资料。

## 容错设计

- 如果没有配置 `GLM_API_KEY`，系统仍然会使用本地模板回答。
- 如果 GLM Chat 调用失败，系统不会直接崩掉，会退回本地模板回答，并提示模型调用暂不可用。
- 真实 Key 仍然只保存在本地 `.env`，不写入开发日志。
